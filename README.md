# Free Space? Oh Yeah!

`fsoy` is a C++20 CLI prototype for HARE v1 file-to-video storage. It encodes an input file into H.264 MP4 frames using Channel A luma-only 4x4 luma blocks and decodes those videos back to the original bytes with SHA-256 verification.

> Prototype status: the current encode/decode path can round-trip videos generated by this prototype, but full rateless/Tornado ECC recovery, executable checkpoint resume, and true multi-volume splitting are still TODOs.

## Requirements

### Required for all encode/decode commands

- A C++20 compiler.
- CMake 3.16 or newer.
- `ffmpeg` on `PATH`; `fsoy` shells out to FFmpeg for H.264 MP4 encode/decode.

### Optional for the NVIDIA GPU Suite

- NVIDIA GPU and driver for runtime acceleration.
- FFmpeg built with NVIDIA support for `--gpu encode`, `--gpu decode`, or `--gpu auto` video-codec acceleration.
- CUDA/NVCC if you want to compile the CUDA frame-synthesis implementation. Without CUDA/NVCC, CMake builds a small NVIDIA driver-detecting stub library so the project still builds on CPU-only systems.

## Build commands

### Default build: CPU executable plus enabled GPU suites

```bash
cmake -S . -B build
cmake --build build
```

Purpose: builds the `fsoy` CLI and, by default, the currently enabled optional GPU suite libraries. Today that means the NVIDIA GPU Suite.

Requirements: CMake, C++20 compiler, and FFmpeg at runtime. CUDA is optional; the NVIDIA suite falls back to a stub if CUDA/NVCC is not available.

### CPU-only build: disable all GPU suites

```bash
cmake -S . -B build -DFSOY_BUILD_GPU_SUITES=OFF
cmake --build build
```

Purpose: builds only the CPU-capable `fsoy` executable and disables all optional GPU suite libraries.

Use this when: you want the smallest/simple build, you are on a machine without GPU drivers, or you want to guarantee no GPU suite library is produced.

### Disable only the NVIDIA GPU Suite

```bash
cmake -S . -B build -DFSOY_BUILD_NVIDIA_GPU_SUITE=OFF
cmake --build build
```

Purpose: keeps the overall suite system enabled, but disables the NVIDIA-specific suite.

Use this when: future non-NVIDIA suites exist and you want those, but not NVIDIA.

### Backward-compatible build options

Older local scripts may still use `FSOY_BUILD_GPU_PLUGINS` or `FSOY_BUILD_NVIDIA_GPU_PLUGIN`. Those are accepted as aliases for `FSOY_BUILD_GPU_SUITES` and `FSOY_BUILD_NVIDIA_GPU_SUITE`.

## Encode commands

### CPU encode

```bash
./build/fsoy encode archive.7z -o archive.mp4
```

Purpose: encodes any input file into a prototype HARE v1 MP4 using CPU frame synthesis and CPU FFmpeg `libx264` H.264 encoding.

Requirements: `ffmpeg` on `PATH` with `libx264` support.

GPU conditions: no GPU suite is required. This remains available even when all GPU suites are disabled at build time.

Output: `archive.mp4` and `archive.mp4.checkpoint`.

### NVIDIA GPU Suite auto encode

```bash
./build/fsoy encode archive.7z -o archive.mp4 --gpu auto
```

Purpose: attempts the NVIDIA GPU Suite acceleration paths during encode. It may use the suite library for luma frame synthesis and asks FFmpeg to use NVIDIA NVENC for H.264 encoding.

Requirements: NVIDIA suite library available for frame synthesis acceleration; FFmpeg with NVENC support for GPU H.264 encoding.

Fallback behavior: if the suite library is missing/unavailable, frame synthesis falls back to CPU. If NVENC fails, the command retries with CPU `libx264`.

Disable conditions: build with `-DFSOY_BUILD_GPU_SUITES=OFF`, build with `-DFSOY_BUILD_NVIDIA_GPU_SUITE=OFF`, omit the suite library at runtime, or use `--gpu off` / no `--gpu` flag.

### NVIDIA GPU Suite frame-synthesis encode

```bash
./build/fsoy encode archive.7z -o archive.mp4 --gpu frames --gpu-plugin ./build/libfsoy_gpu_nvidia.so
```

Purpose: uses the NVIDIA GPU Suite shared library only for luma frame synthesis. H.264 video encoding remains CPU `libx264`.

Requirements: the NVIDIA suite shared library (`libfsoy_gpu_nvidia.so`, `fsoy_gpu_nvidia.dll`, or `libfsoy_gpu_nvidia.dylib`) and an available NVIDIA driver/device for the suite implementation to report available.

Fallback behavior: if the suite cannot load or reports unavailable, frame synthesis falls back to CPU.

Disable conditions: do not pass `--gpu frames`; build without the NVIDIA suite; remove the suite library; or pass `--gpu off`.

### NVIDIA GPU Suite NVENC encode

```bash
./build/fsoy encode archive.7z -o archive.mp4 --gpu encode
```

Purpose: asks FFmpeg to use NVIDIA NVENC (`h264_nvenc`) for H.264 compression. This is the encode path most likely to provide a major speedup when video compression is the bottleneck.

Requirements: NVIDIA GPU/driver and an FFmpeg build with `h264_nvenc` support.

Fallback behavior: if FFmpeg/NVENC fails, `fsoy` retries the encode with CPU `libx264`.

Disable conditions: omit `--gpu encode`, use `--gpu off`, or use an FFmpeg build without NVENC support.

## Decode commands

### CPU decode

```bash
./build/fsoy decode archive.mp4 -o restored.7z
```

Purpose: decodes one or more prototype HARE v1 MP4 files back to the original file and verifies SHA-256.

Requirements: `ffmpeg` on `PATH`.

GPU conditions: no GPU suite is required. This remains available even when all GPU suites are disabled.

### NVIDIA GPU Suite CUDA/NVDEC decode

```bash
./build/fsoy decode archive.mp4 -o restored.7z --gpu decode
```

Purpose: asks FFmpeg to use CUDA/NVDEC hardware video decode while `fsoy` still performs HARE block extraction and SHA-256 verification on CPU.

Requirements: NVIDIA GPU/driver and an FFmpeg build with CUDA/NVDEC hardware acceleration support.

Fallback behavior: if CUDA/NVDEC decode fails, `fsoy` retries that input with CPU FFmpeg decode.

Disable conditions: omit `--gpu decode`, use `--gpu off`, build/run without NVIDIA support, or use an FFmpeg build without CUDA/NVDEC support.

### Decode multiple parts

```bash
./build/fsoy decode archive.part001.mp4 archive.part002.mp4 -o restored.7z
```

Purpose: decodes multiple input video parts in order.

Current limitation: true multi-volume splitting is still TODO, so this is CLI plumbing for the intended workflow rather than a complete split/resume implementation.

## Checkpoint commands

### Encode with recorded output-size limit

```bash
./build/fsoy encode archive.7z -o archive.mp4 --max-output-size 4G
```

Purpose: records the requested maximum output size in the scaffolded checkpoint flow.

Current limitation: the v1 scaffold still writes one feed volume; true split-at-size behavior is not implemented yet.

### Resume encode from checkpoint

```bash
./build/fsoy encode --resume archive.mp4.checkpoint
```

Purpose: intended future resume path after interruption.

Current limitation: recognized by the CLI, but not executable yet.

### Decode via checkpoint

```bash
./build/fsoy decode --checkpoint archive.mp4.checkpoint -o restored.7z
```

Purpose: intended future decode path using checkpoint metadata.

Current limitation: recognized by the CLI, but not executable yet.

## NVIDIA GPU Suite model

The GPU acceleration system is classified into vendor/architecture suites. The active suite right now is the NVIDIA GPU Suite.

The core executable owns CLI parsing, file I/O, HARE packing/extraction, FFmpeg pipe management, checkpoint writing, fallback handling, and SHA-256 verification. The NVIDIA GPU Suite is supplementary and can accelerate specific stages:

- `--gpu frames`: suite library accelerates luma frame synthesis.
- `--gpu encode`: FFmpeg/NVENC accelerates H.264 video encoding.
- `--gpu decode`: FFmpeg/CUDA/NVDEC accelerates H.264 video decoding.
- `--gpu auto`: attempts the available NVIDIA encode/decode acceleration paths relevant to the selected command, with CPU fallback.

The suite library is loaded from an explicit `--gpu-plugin` path, the `FSOY_GPU_PLUGIN` environment variable, or default platform names such as `libfsoy_gpu_nvidia.so`, `fsoy_gpu_nvidia.dll`, or `libfsoy_gpu_nvidia.dylib`. Older names such as `libfsoy_nvidia_gpu.so` are still checked for compatibility.

## Expected GPU impact

- Frame synthesis (`--gpu frames`): usually a small-to-moderate win because the current work is simple 4x4 block filling and still copies frames through CPU memory into FFmpeg.
- Encode (`--gpu encode`): potentially substantial when CPU `libx264` is the bottleneck and NVENC is available.
- Decode (`--gpu decode`): potentially helpful when H.264 decode is the bottleneck, but HARE block extraction and SHA-256 verification still run on CPU.

Exact speedups must be benchmarked on the target machine, FFmpeg build, driver version, input size, and selected bitrate/quality settings.

## HARE v1 choices in this prototype

- Channel A only: one bit per 4x4 luma block, thresholded as black/white.
- 1920x1080, 24 fps, H.264 Main Profile MP4 through FFmpeg.
- Streaming feed to FFmpeg; the full set of video frames is not materialized as image files.
- File-level SHA-256 verification on decode.
- GPU support is optional and supplementary; CPU encode/decode remains available without any GPU suite.

## Current limitations

The repository now contains the runnable scaffold for v1, but full rateless/Tornado ECC recovery, executable checkpoint resume, and true multi-volume splitting are still implementation TODOs. The current encode/decode path can round-trip files through an MP4 generated by this prototype and verify SHA-256, but it is not yet the final YouTube-resilient HARE v1 implementation because ECC recovery is not complete.
